Use of the protocol, which is designed to facilitate the download of data from existing user systems, removes the requirement to submit information via our existing document templates or online forms, which often have to be manually keyed in by regulated parties at point of data entry.

Background

The Data Exchange Protocol (DEP) offers a straightforward and effective way to share information produced by regulated parties. It does this by focusing on transferring the data values themselves from a customer's existing data systems via the export of a standardised text file ('csv' or 'xml') or a very simple column based spreadsheet. This, which is available across a number of different 'Return Types', replaces the need to complete traditional ‘forms’. While many are now provided in electronic format,  still generally revolve around manual completion of something similar to a paper based product. The methodology also includes the ability for customers to ‘push’ data to us using automated file transfers, as opposed to sending individual emails or uploading files manually through our webpage - though these more traditional methods will also remain available. The Data Exchange Protocol can offer the following benefits

  • Delivers significant burden reduction for regulated business
  • Introduces a more flexible approach with both internal and external benefits
  • Delivers significant cost savings
  • Provides improved data quality
  • Has relatively small set up cost and a short payback period

How it Works

The vast majority of regulatory compliance monitoring data that we require regulated parties to submit, once formatting and presentational aspects have been removed, has a similar structure. This type of data is generally submitted against a permit or other ‘Unique EA Reference’ against which we already hold other information in our permitting or registration systems e.g. the name and address of the operator and of the site/location in question, the company number and SIC code. The data also has a monitoring point or sampling location, which could include ‘the gate’ or the ‘site’ as a whole, which likewise, we already hold information about from data gathered through Permitting. The monitoring reading has a date (and time) or period to which it relates. It has a parameter, a value, a unit and some other pieces of information depending on the type of return. 

Many customers have software which store information in the above way, (monitoring point, date, parameter, unit, value) and by outputting it and submitting it to us within the structure set out below, the Environment Agency will be able to 'mail merge' it to any relevant presentational layout and also transform the data into a format which can be uploaded directly to our data systems using their existing upload mechanisms. e.g. transforming the data into pre-existing xml or Excel spreadsheet formats.

The approach allows operators to ‘dump’ data from their information systems for all or part of their regulatory estate across a number (or all) of their ‘Return Types’. Information is supplied in the same simple format, which means operators can choose how they breakdown their data submission to match their own internal structures and data systems, rather than matching ours. This provides much more flexibility and opportunity for cost savings. The removal of the need to maintain and use expensive, macro based Excel spreadsheet tools and the costs associated with consultation on changes to standard forms  will also provide benefits for the Environment Agency and those we regulate.

The Data Exchange Protocol

The data exchange protocol comprises four elements:

  • A generic file output specification defining the data elements to be included
  • A set of data standards or controlled lists defining allowable values for those elements across all regulatory regimes
  • Regime Specific Rules defining how particular information should be mapped to the generic data elements and providing further refinement of over arching data standards for that regime. These are further divided into Return Type Rules.
  • A mechanism for uploading data to the Environment Agency via a file sharing or file transfer protocol

File Output Specification

Data Exchange Files can be output from existing data systems as comma separated values (csv), XML (specific xml schema to follow) or a very simple column based spreadsheet. The data elements are given below with a description, an example and an indication of a data standard or a reference to one of the Data Standards included later in the document.  This is a generic specification that has been designed to cover a number of different regulatory regimes that require compliance data. Some of the data elements given below are not relevant for every regime or every sample type, for instance Monitoring End Date would only be relevant where the measurement covers a period of time, e.g. monthly water use,  and is superfluous for spot samples. Other data elements are always required, for instance the 'Return Type' which enables us to identify the requirement against which the data has been submitted and an 'Unique EA Identifier', for instance a Permit Reference, which enables us to identify information about the data element, such as the site and operator name to which it relates, without the need to specifically include that in full, in the output itself.  Regime Specific Rules set out what should be provided in each case.

A single output file may contain rows of data covering more than one site or permit and/or covering more than one return type.

Data Standards

The Environment Agency is producing documented data standards across a number of common data reporting regimes. Data standards developed so far, that apply to the Data Exchange Protocol are:

  • Date & Time: This data standard tells us how to format and work with Date and Time information.

    Use of this data standard results in dates and times being consistently formatted.  This consistency increases completeness, clarifies the date/time precision, reduces chance of error, removes ambiguity, reduces uncertainty and increases the interoperability of our data and systems.

  • Units: This data standard presents approved units and measures we may need in our work.

    Use of this data standard results in more consistent use of units and decreased instances of attribute and reporting inconsistencies and errors which promotes clarity and interoperability of our data and systems.

  • Symbols: This data standard presents approved symbols we may need in our work.

    Use of this data standard results in more consistent use of Symbols, improving clarity and decreasing ambiguity

  • Parameter and Substance Names:

    We have started some work on a list of parameter and substance names. However the sheer number of different parameters and substances and the various existing ways for categorising them, means it will be sometime before this list is finalised. In the meantime, the section of Appendix 1 which covers this aspect lists some naming conventions for physical parameters and a small number of substances. For other substances and chemicals you are currently asked to cross refer with other existing lists, publically available lists.

Use of this data standard results in fewer errors, removes ambiguity and allows better processing of received data across a number of different return types

Please Note

In some cases the data standards given are aspirational and an indicator of future constraints on acceptable information.  Data formats outside these standards may be allowed for certain regimes for an interim period. Please see 'Return Specific Rules' will give more information on this.

Data Submission via a File Transfer Protocol

A file transfer utility has been set up to allow our customers to upload data to us without the need to submit files via email. This enables large files to be transmitted (up to 10GB, whereas existing restrictions prevent files over 10MB being received by email and files over 5MB are subject to delays) and it also allows for faster and easier submission.

The system we currently use to provide this service is Sharefile by Citrix. Sharefile has 256 bit encryption and is very secure. Sharefile offers a number of desktop utilities that make submission of your data a very simple process. There is no requirement for a customer to have their own Sharefile licence to submit data to us, as this is covered by our user licence. For more information about Sharefile go to the their website

Appendix 1 - Data Standards

Date and Time

This tells us how to format and work with Date and Time information.

Character Definitions The following character definitions are used in this Quick Guide:
  • [Y] represents a digit used in the time element “year”;
  • [M] represents a digit used in the time element “month”;
  • [D] represents a digit used in the time element “day”;
  • [h] represents a digit used in the time element “hour”;
  • [m] represents a digit used in the time element “minute”;
  • [s] represents a digit used in the time element “second”;
  • “T” represents the start of the time component of a recorded date and time.
Date Format A date is recorded as YYYY-MM-DD where:
  • [YYYY] is the full 4-digit year;
  • [MM] is the 2-digit calendar month number from 01 to 12, with a leading “0” inserted for the months of January to September e.g. “01” for January;
  •  [DD] is the calendar day from 01 to 31 (01 to 30 for April, June, September and November and 01 to 28 for February with 29 in a leap year) with a leading “0” inserted for days before the 10th day of a month e.g. “02” for 2 January;
  • The separator character is a hyphen “-“.
  • Examples: 3 September 1972 is recorded as 1972-09-03.
Time Format A time is recorded as hh:mm:ss where:
  • [hh] is the full 2-digit hour using the 24-hour clock that runs from 00 to 23 with a leading “0” inserted for the hours before 10am e.g.“07” for 7am.
  • [mm] is the full 2-digit minute that runs from 00 to 60 with a leading “0” inserted for the minutes before 10 minutes e.g.“06” for 6 minutes;
  • [ss] is the full 2-digit second that runs from 00 to 60, with a leading “0” inserted for the seconds before 10 seconds e.g.“02” for 2 seconds;
  • The separator character is a colon “:“.
  • Examples: Four seconds past 7:13am is recorded as 07:13:04.  Ten seconds past 11:04pm is recorded as 23:04:10.
Date and Time Format Date and Time are recorded together as YYYY-MM-DDThh:mm:ss (although by agreement of both parties in data exchange, the “T” may be dropped and replaced with a single space i.e. YYYY-MM-DD hh:mm:ss) where:
  • Date is recorded as above;
  • Time is recorded as above;
  • A “T” is inserted between the Date and Time.
  • Examples: Four seconds past 7:13am on 3 September 1972 is recorded as 1972-09-02T07:13:04.  Ten seconds past 11:04pm on 16 December 2013 is recorded as 2013-12-16T23:04:10.
Reduced Precision Formats

If a precise Date and Time is not required (or is not actually known), the full Date and Timeformat can be amended by omitting digits and characters, starting from the extreme right-hand side.

This has the effect of reducing the precision of the recorded Date and Time.

Examples:

Situation Action Example
If you need the exact date and time to the nearest second Record full date and time 2013-09-03 12:01:56
If you need the date and time to the nearest minute Omit seconds from the time element 2013-09-03 12:01
If you need the date and time to the nearest hour Omit seconds from the time element 2013-09-03 12
If you need the date to the nearest day Omit the time element 2013-09-03
If you need the date to the nearest month Omit the day and time elements 2013-09
If you need the year Omit the month, day and time elements 2013
Date and Time Range

For a Date and Time range, use the Date and Timeformat above for the time of the start interval and again for the end interval with a forward slash “/” as a separator character i.e.

YYYY-MM-DDThh:mm:ss/YYYY-MM-DDThh:mm:ss Reduced precision formatscan also be used in Date and Time ranges.

Examples:

Situation Action Example
If you need the exact date and time range to the nearest second Record full dates and times with the slash separator 2013-09-03T12:01:56/2014-09-04T 12:30:04
If you need the date and time range to the nearest minute Omit seconds from the time element 2013-09-03T12:01/2014-09-04T12:30
If you need the date range to the nearest day Omit the time element 2013-09-03/2014-09-04
If you need the date range to the nearest month Omit the day and time elements 2013-09/2014-09
If you need the range from year to year Omit the day, month and time elements 2013/2014
